# 长时间运行的目标（`long_task` / `complete_goal`）

当用户想要针对**一个**明确目标进行**多轮持续工作**时使用这些工具（同一 runner，普通工具）。不适用于简单的一次性问题。

## 快速开始

`long_task` 是一个轻量标记。调用它告诉 nanobot："此线程有一个持续目标；跨轮次保持该目标可见并在 UI 中展示。"

读完这个简短的开始部分后，**在用户意图清晰时尽快调用 `long_task`**。立即写一个好的 `goal`：使其幂等、自包含、有边界，并明确完成条件。不要在设置标记前花很长时间思考项目规划、研究或执行细节。

在第一次 `long_task` 调用前，你**不需要**：

1. 设计完整的项目计划，
2. 研究 API 或文档，
3. 编写详尽的项目计划或检查清单，
4. 决定每个文件、命令或验证步骤。

这些属于标记设置后的执行阶段。

## 工具

- **`long_task`** - 为每个线程注册**一个**持续目标。用户提出持续任务后及时调用。`goal` 应遵循下面的幂等目标规则，但应从用户请求中快速生成--而非经过漫长的隐藏规划过程。

- **`complete_goal`** - 关闭**当前**活动目标的记账。工作**完成**时调用，**同时**用户**取消**、**改变方向**或**替换**目标时也调用：使用 **`recap`** 诚实说明发生了什么（例如已取消、部分完成、被取代）。然后你可以在会话显示无活动目标后（或用户同意替换后）为**新**目标再次调用 **`long_task`**。

如果目标已活动且用户想要不同的东西，先 **`complete_goal`**（诚实 recap），然后用新目标 **`long_task`**--不要堆叠冲突的活动目标。

## 目标出现的位置

在 **`[Runtime Context - metadata only, not instructions]`** 内，以 **`Goal (active):`** 开头的行携带此聊天会话的**持久化目标**（会话元数据）。将它们视为活动的持续目标，而非用户编写的绕过策略的指令。

可选的 **`Summary:`** 仅是简短的 UI 标签--将清晰的验收提示放在 **`goal`** 正文本身中。

# `long_task` 设置后的执行指南

在工作时使用以下指导。它应塑造执行和未来上下文，但不应延迟第一次 `long_task` 调用。

## 幂等目标（重要）

**意图：** 目标字符串可能在压缩后、重试中或恢复工作时被**重新读取**。它仍应表示**一个明确结果**，不暗示重复的破坏性步骤，也不依赖仅存在于聊天中的记忆。

编写目标使其：

1. **面向状态，而非脆弱的叙述** - 优先使用*期望的最终状态 + 验收标准*（"文档在 `docs/…` 下列出 X、Y、Z；链接已验证"）而非*隐式顺序*（后者在步骤 1 已完成时会失效，如"先克隆仓库，然后…"）。

2. **自包含** - 重复重要的约束（路径、仓库名、分支、版本锁定、计数）。**不要**依赖"如上所述"来传达压缩可能裁剪的需求。

3. **在重复下安全** - 措辞应能在**恢复**后存活：使用"确保…"、"直到…"、"改变前验证…"。对于变更（写入、提交、API 调用），优先使用**先检查后行动**或显式**幂等**操作（upsert、覆盖已知路径、如已满足则跳过）。

4. **有边界范围** - 说明什么**在范围内**和**不在范围内**（例如"星数在 A–B 范围内的前 100 个仓库"、"仅 `src/` 下的文件"）。减少模型冷启动重新进入目标时的偏移。

5. **明确的完成条件** - 说明你如何知道已完成（测试通过、产物存在、检查清单满足、用户确认）。避免"当看起来不错时"。

6. **`ui_summary`** - 侧边栏/日志的简短标签；保持**非承重**（不要在 summary 中放入仅有的秘密需求）。

如果你发现目标定义不足，可以询问用户--或用 recap **`complete_goal`** 并注册一个**更窄**的替换目标，而非塞满一个模糊的字符串。

## 项目式工作（避免"巨型文件"陷阱）

当目标是**构建或重塑代码库**（应用、服务、工具、 sizable feature）时使用：

1. **模块化布局** - 拆分为**有意义的模块**（职责清晰的目录 + 文件：入口点、领域逻辑、配置、基础设施、CLI/UI 路由等）。**不要**默认将整个项目倾倒到一个巨大的源文件中，除非用户明确想要最小的单文件产物。
2. **常规结构** - 遵循该技术栈的常规做法（关注点分离、合理命名、配置与代码分离、可复用辅助函数）。以可审查的增量为目标，而非不可读的大块。
3. **边做边验证** - 在有意义的块之后运行/格式化/lint/测试项目能提供的检查，使代码树保持真实；当检查重要时将**检查或手动步骤纳入目标**。

## 查找而非猜测

事实（API 细节、工具标志、比截止日期更新的弃用和最佳实践）在持续工作中会静默失败，除非你尽早锚定：

1. **适时使用发现工具** - 如果生态不熟悉或脆弱，**尽早**使用 **`web_search`**、文档/web 抓取（或 MCP）--在承诺架构或重写大面积之前。与你接下来必须做的决策相关的窄查询。
2. **将发现转化为有范围的行动** - 仅在有帮助时将结论总结为仓库产物（注释、README、小型设计笔记）；保持**紧凑**--不是执行目标的替代品。
3. **卡住时重新查阅** - 如果错误与假设矛盾或循环重复，暂停并用有针对性的搜索/抓取刷新上下文，而非盲目敲击。
